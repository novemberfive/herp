# 轻量级独立部署版非功能需求说明

## 1. 文档定位

### 1.1 适用范围
- 本文档适用于 [`C:\Work\projects\herp\plan\pd\轻量级独立部署架构设计文档.md`](C:\Work\projects\herp\plan\pd\轻量级独立部署架构设计文档.md) 对应的单机部署版本。
- 默认部署形态为 `Spring Boot + MySQL + Vue` 单体应用，运行在 1 台办公电脑或低配服务器上，供局域网内少量用户使用。
- 非本文档目标：微服务拆分、IoT 海量设备接入、多院区多租户、复杂工作流编排、消息中间件和高可用集群。

### 1.2 验收分层
| 分层 | 含义 | 当前要求 |
|------|------|----------|
| `P0` | 上线必需 | 必须满足，直接影响单机版交付 |
| `P1` | 建议具备 | 推荐满足，可在后续迭代补齐 |
| `P2` | 扩展能力 | 仅保留设计约束，不纳入当前完成率 |

### 1.3 当前验收前提
- `P0` 验收以主流程可用、单机可部署、数据可回滚、异常可重试为优先。
- 若 `plan/req`、`plan/test`、`plan/ux` 中出现与微服务、Redis、MQ、IoT 强绑定的条目，除明确标注为 `P0` 的最小替代方案外，统一视为 `P2`。
- 当前版本通知能力优先采用页面状态、列表提示、定时扫描和备注留痕，不要求短信、邮件、Webhook 或消息队列。

## 2. 性能需求

### 2.1 响应时间
| 业务场景 | 目标值 | 分层 | 说明 |
|---------|--------|------|------|
| 登录、菜单切换 | `<= 2s` | `P0` | 局域网常规使用 |
| 常规列表查询 | `<= 3s` | `P0` | 含分页和常见筛选 |
| 单据保存/提交 | `<= 3s` | `P0` | 采购、验收、入库、领用、借用等 |
| 统计报表查询 | `<= 5s` | `P1` | 基础统计、折旧、处置报表 |
| 预警扫描任务 | `<= 60s/轮` | `P1` | 允许定时任务批处理 |
| 批量导入/导出 | `<= 10s/1000 条` | `P1` | 当前规模下的经验目标 |

### 2.2 并发与规模
| 指标 | 目标值 | 分层 | 说明 |
|------|--------|------|------|
| 同时在线用户 | `<= 20` | `P0` | 典型值 5-10 |
| 峰值请求吞吐 | `<= 20 QPS` | `P0` | 单机局域网环境 |
| 资产记录规模 | `<= 5000` | `P0` | 超出后需重新评估部署形态 |
| 附件单文件大小 | `<= 10MB` | `P0` | 以图片、PDF 为主 |
| 历史数据保留 | `>= 3 年` | `P1` | 超长期归档可后置 |

### 2.3 资源约束
- 推荐运行环境：`2C4G + 40GB SSD + MySQL 8.0`。
- 最低可接受环境：`1C2G + 20GB SSD`，允许报表和导入速度下降。
- 不要求横向扩展；单机资源不足时优先升级 CPU、内存、磁盘。

## 3. 可靠性与运维需求

### 3.1 可用性
| 指标 | 目标值 | 分层 | 说明 |
|------|--------|------|------|
| 服务可用性 | 工作时间内稳定可用 | `P0` | 允许短暂停机后人工重启 |
| 一般故障恢复时间 | `<= 30 分钟` | `P0` | 以重启服务和检查数据库为主 |
| 严重故障恢复时间 | `<= 4 小时` | `P1` | 依赖备份恢复 |
| 部署重启复杂度 | `<= 3 步` | `P0` | 启动应用、检查数据库、验证登录 |

### 3.2 数据可靠性
- 所有核心写操作必须使用本地数据库事务，保证 ACID。
- 失败写入不得产生半成功脏数据；若失败，应允许用户重新提交。
- 关键单据应保留状态、审批意见、更新时间和操作痕迹，便于人工核对。
- 后端异常应返回统一错误结构，前端给出明确提示，不得静默失败。

### 3.3 备份与恢复
| 项目 | 要求 | 分层 |
|------|------|------|
| 全量备份 | 每天至少 1 次 | `P0` |
| 手工备份导出 | 支持数据库或文件目录手工备份 | `P0` |
| 备份保留周期 | `>= 7 天` | `P0` |
| 异地/云备份 | 可选 | `P1` |
| 自动恢复演练 | 建议每月 1 次 | `P1` |

### 4.1 认证与会话

| 单点登录 | `P2` | 当前不作为单机版前置 |

### 4.2 权限控制
| 要求 | 分层 | 说明 |
|------|------|------|
| 登录后访问控制 | `P0` | 未登录不得访问受保护路由/接口 |
| 基础角色字段区分 | `P0` | 支持最小角色识别 |
| 用户/角色/RBAC 管理页面 | `P1` | 当前已完成用户、角色和菜单权限基础版 |
| 菜单级权限 | `P1` | 当前已完成前端菜单/路由基础控制 |
| 按钮级权限 | `P1` | 当前工作树已完成多模块首轮收口，后续以查漏补缺为主 |
| 部门数据权限 | `P1` | 已完成当前轻量级单机版验收基线：资产/报表、采购、验收、入库、领用/借用/调拨、盘点结果按部门范围收口，盘点计划/任务按 `create_user_id` 实施轻量级可见范围控制 |
| 细粒度审计追踪 | `P1` | 当前已落地后端核心业务留痕（登录/登出、用户/角色、基础主数据、资产卡片、采购/验收/入库、领用/借用/调拨、维修、处置、盘点写操作），并已补齐审计查询 API/UI；后续主要继续扩展数据权限范围 |

### 4.3 数据与网络安全
- 内网部署时允许 HTTP 运行，但若跨机器访问或连接不可信网络，建议启用 HTTPS。
- 密码必须采用不可逆加密存储，禁止明文落库。
- 导出、附件下载、敏感字段展示可先采用最小可用控制，不要求当前引入审批水印体系。
- 不要求单机版默认配置 WAF、IDS/IPS、DDoS 防护，这些统一视为外围基础设施能力。

## 5. 易用性与界面要求

### 5.1 终端范围
| 终端 | 要求 | 分层 |
|------|------|------|
| PC 管理端 | `1280px+` 稳定可用 | `P0` |
| PC 员工端 | 查询、申请、待办可用 | `P0` |
| 手机/H5 | 门户查询和简单表单可用 | `P1` |
| 离线移动盘点 | 扩展设计 | `P2` |

### 5.2 交互一致性
- 表单必填、错误提示、保存成功、删除确认、空状态提示需全局一致。
- 主流程页面优先接入真实主数据选择器，减少单机部署中的自由文本录入差错。
- 列表页默认提供分页、常用筛选、重置和刷新能力。
- 错误提示需指向可操作原因，如“资产不存在”“状态不允许提交”“库存不足”。

### 5.3 可培训性
- 新用户在 30 分钟内应能完成登录、查询、采购申请、验收、入库、领用等基本操作培训。
- 系统应保留最小帮助信息，如字段占位提示、空状态指引、操作按钮命名一致。

## 6. 可维护性与工程要求

### 6.1 构建与发布
| 要求 | 分层 | 说明 |
|------|------|------|
| 后端可执行编译通过 | `P0` | `mvn -q -DskipTests compile` |
| 前端生产构建通过 | `P0` | `npm run build` |
| 环境配置可区分 | `P0` | 至少支持开发/生产 |
| Docker/CI/CD | `P1` | 当前可后置 |

### 6.2 日志与监控
- 必须输出基础运行日志和错误日志，便于人工排查。
- 关键定时任务、审批状态变更、批量导入导出失败应有日志记录。
- 高级监控平台、集中式日志系统、链路追踪统一视为 `P2`。

### 6.3 可测试性
- `P0` 以手工回归和最小构建验证为主。
- `P1` 应逐步补齐后端单元测试、接口测试和前端关键页面测试。
- 当前未建立自动化测试目录不阻塞上线判断，但应作为后续工程治理任务持续推进。

## 7. 兼容性与集成边界

### 7.1 兼容性
- 浏览器：`Chrome 80+`、`Edge 80+` 为主，其他现代浏览器兼容为 `P1`。

## 8. 当前版本结论

- 当前轻量级独立部署版的非功能验收基线已经收敛为：单机可部署、局域网可访问、主流程可用、构建可通过、数据可备份恢复。
- 不再以高并发、微服务、高可用集群、IoT 海量定位或复杂外部集成为当前上线前置条件。
- 后续若系统规模突破本文档边界，应重新评估并切换到更完整的平台型架构文档。
