# 轻量级独立部署架构设计文档（村卫生室/小型机构版）

## 1. 文档概述

### 1.1 适用场景
本架构专为**数据量小、并发低、容错率高**的微型机构设计，典型场景包括：
- 村卫生室（资产数量 < 500，用户数 < 10）
- 小型诊所
- 社区服务站
- 个体药店
- 小型实验室

### 1.2 设计原则
- **极简主义**：能少则少，去除一切非必要组件
- **单体架构**：所有功能打包为一个可执行文件
- **零依赖部署**：除数据库外，不依赖任何中间件（无 Redis、无 MQ、无 ES）
- **低成本**：可在低配服务器（1核2G）或普通PC上运行
- **易维护**：非技术人员经过简单培训即可完成部署和重启

### 1.3 与非功能需求的适配说明
| 原非功能需求 | 轻量级适配方案 |
|------------|--------------|
| 高可用 (99.9%) | 降级为"可接受短暂停机"，宕机后重启即可，数据不丢失 |
| 高并发 (1000 QPS) | 降级为支持 20 QPS，满足 10 人同时操作 |
| 响应时间 < 200ms | 降级为 < 2s，本地局域网访问通常 < 500ms |
| 横向扩展 | 不支持，采用纵向扩展（升级单机配置） |
| 分布式事务 | 不适用，使用本地事务 |
| 消息队列 | 移除，改用定时任务轮询 |
| 多级缓存 | 简化为应用层内存缓存（Caffeine/Guava） |

## 3. 技术栈选型（精简版）

### 3.1 整体技术栈

| 层级 | 技术选型 | 版本 | 说明 |

| **UI 框架** | Element Plus | 2.4+ | 组件丰富，开箱即用 |
| **HTTP 客户端** | Axios | 1.6+ | 标准 HTTP 请求 |
| **状态管理** | Pinia | 2.1+ | 比 Vuex 更轻量 |
| **扫码库** | html5-qrcode | 2.3+ | 纯 JS 实现，无需原生 App |
| **后端框架** | Spring Boot | 3.2+ | 单体应用，内嵌 Tomcat |

| **数据库** | MySQL | 8.0+ | 稳定可靠，社区版免费 |
| **连接池** | HikariCP | 5.0+ | Spring Boot 默认，高性能 |
| **本地缓存** | Caffeine | 3.1+ | 进程内缓存，替代 Redis |

| **日志** | Logback | 内置 | Spring Boot 默认 |
| **文档** | Knife4j (Swagger) | 4.3+ | 接口文档在线查看 |

### 3.2 移除的组件对比

| 原微服务架构组件 | 轻量级替代方案 | 原因 |
|----------------|--------------|------|
| Nginx 负载均衡 | 不需要 | 单实例，无流量分发需求 |
| Spring Cloud Gateway | 不需要 | 无微服务路由需求 |
| Nacos/Eureka | 不需要 | 无服务注册发现需求 |
| Redis | Caffeine 本地缓存 | 数据量小，单机内存足够 |
| RabbitMQ/Kafka | Spring Schedule 定时任务 | 无高并发异步需求 |
| Elasticsearch | MySQL LIKE 查询 | 数据量 < 10 万，LIKE 足够快 |
| MinIO/OSS | 本地文件系统 | 图片数量少，直接存磁盘 |
| Prometheus + Grafana | Spring Boot Actuator + 简单日志 | 无需复杂监控 |
| SkyWalking | 不需要 | 无分布式链路追踪需求 |
| Kubernetes | 不需要 | 单机部署，无需容器编排 |

### 3.3 服务器配置建议

#### 最低配置（勉强运行）
- CPU: 1 核
- 内存：2 GB
- 硬盘：20 GB SSD
- 操作系统：Windows 10 / Ubuntu 20.04
- 适用：资产数量 < 200，用户数 < 5

#### 推荐配置（流畅运行）
- CPU: 2 核
- 内存：4 GB
- 硬盘：40 GB SSD
- 操作系统：Windows 10 / Ubuntu 22.04
- 适用：资产数量 < 1000，用户数 < 20

#### 理想配置（充裕余量）
- CPU: 4 核
- 内存：8 GB
- 硬盘：80 GB SSD
- 操作系统：Windows Server 2019 / Ubuntu 22.04 LTS
- 适用：资产数量 < 5000，用户数 < 50

### 5.2 两种部署方式

#### 方式一：JAR 包直接运行（最简单）

**适用场景**：无 Docker 环境，Windows 优先

**步骤**：
1. 安装 JDK 17（或 JDK 21）
2. 安装 MySQL 8.0
3. 创建数据库 `asset_db`
4. 执行 SQL 初始化脚本
5. 修改 `application.yml` 配置数据库连接
6. 运行 `java -jar asset-management.jar`
7. 浏览器访问 `http://localhost:8080`

**开机自启（Windows）**：
```batch
:: 创建 start.bat
@echo off
cd /d %~dp0
java -jar asset-management.jar
```
将 `start.bat` 放入启动文件夹：`shell:startup`

**开机自启（Linux）**：
```bash
# 创建 systemd 服务
sudo nano /etc/systemd/system/asset.service
```

```ini
[Unit]
Description=Asset Management System
After=syslog.target network.target

[Service]
User=root
WorkingDirectory=/opt/asset
ExecStart=/usr/bin/java -jar asset-management.jar
SuccessExitStatus=143
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

```bash
# 启用服务
sudo systemctl enable asset
sudo systemctl start asset
```

#### 方式二：Docker 部署（推荐）

**适用场景**：有 Docker 环境，希望隔离依赖

**docker-compose.yml**：

## 6. 缓存策略（简化版）

### 6.1 本地缓存设计（Caffeine）

由于移除了 Redis，使用 **Caffeine** 作为进程内缓存。

**配置**：
```yaml
# application.yml
caffeine:
  cache:
    spec: maximumSize=1000,expireAfterWrite=10m
```

**缓存场景**：

| 缓存内容 | 缓存策略 | 过期时间 | 最大容量 |
|---------|---------|---------|---------|
| 用户信息 | 登录后缓存 | 30 分钟 | 100 |
| 资产分类树 | 启动时加载 | 2 小时 | 10 |
| 存放地点树 | 启动时加载 | 2 小时 | 10 |
| 字典数据 | 启动时加载 | 永久（手动刷新） | 50 |
| 资产详情 | 查询时缓存 | 5 分钟 | 500 |
| 报表统计数据 | 查询时缓存 | 10 分钟 | 100 |

**代码示例**：
```java
@Configuration
public class CaffeineConfig {
    
    @Bean
    public Cache<String, Object> assetCache() {
        return Caffeine.newBuilder()
                .maximumSize(500)
                .expireAfterWrite(5, TimeUnit.MINUTES)
                .build();
    }
    
    @Bean
    public Cache<String, List<Category>> categoryTreeCache() {
        return Caffeine.newBuilder()
                .maximumSize(10)
                .expireAfterWrite(2, TimeUnit.HOURS)
                .build();
    }
}
```

**缓存更新策略**：
- 数据变更时**主动失效**缓存
- 定时任务**定期刷新**热点数据
- 应用重启后**重新加载**基础数据

### 6.2 为什么不使用 Redis？

| 对比项 | Redis | Caffeine | 选择理由 |
|-------|-------|----------|---------|
| 部署复杂度 | 需单独安装/维护 | 零配置，内置 | 村卫生室无运维人员 |
| 内存占用 | 额外进程占用 | 与应用共享 JVM | 节省资源 |
| 网络开销 | 需网络调用 | 进程内调用，极速 | 单机无需网络 |
| 持久化 | 支持 RDB/AOF | 不支持（重启丢失） | 缓存丢了可重建，不影响业务 |
| 分布式 | 支持多节点共享 | 仅单节点 | 单机部署无需分布式 |
| 成本 | 需额外服务器 | 免费 | 降低成本 |

**结论**：对于数据量 < 5000 条的单机系统，Caffeine 完全够用，且更简单。

### 7.1 原 MQ 场景及替代方案

| 原 MQ 应用场景 | 轻量级替代方案 | 实现方式 |
|--------------|--------------|---------|
| 资产入库后发送通知 | 同步调用 + 定时任务补偿 | 入库成功后直接调通知服务，失败则定时重试 |
| 维修完成后更新状态 | 同步更新 | 维修单审核通过后立即更新资产状态 |
| 盘点任务分配 | 同步写入数据库 | 创建盘点单时直接插入明细表 |
| 临期资产预警 | 定时任务每日扫描 | Spring Schedule 每天 8:00 执行 |
| 异步生成报表 | 同步生成 + 进度条 | 小数据量可直接同步生成，前端显示 loading |
| 批量导入异步处理 | 同步处理 + 分批次 | 每批 100 条，前端显示进度 |
| 操作日志记录 | 异步线程池 | 使用 `@Async` 写入日志表 |
| 短信/邮件通知 | 同步调用第三方 API | 直接调用阿里云短信 API |

### 7.2 定时任务设计

**配置**：
```yaml
# application.yml
task:
  enabled: true
  warning:
    cron: "0 0 8 * * ?"  # 每天 8:00 执行
  cleanup:
    cron: "0 0 2 * * ?"  # 每天 2:00 执行
```

**任务代码**：
```java
@Component
@Slf4j
public class WarningTask {

    @Autowired
    private AssetCardService assetCardService;
    
    @Autowired
    private NotificationService notificationService;

    /**
     * 每日检查临期资产（维保到期、检定到期、报废到期）
     */
    @Scheduled(cron = "${task.warning.cron}")
    public void checkExpiringAssets() {
        log.info("开始执行临期资产检查任务...");
        
        LocalDate today = LocalDate.now();
        LocalDate warningDate = today.plusDays(30); // 提前 30 天预警
        
        // 查询即将到期的资产
        List<AssetCard> expiringAssets = assetCardService.findExpiringAssets(warningDate);
        
        for (AssetCard asset : expiringAssets) {
            String warningType = determineWarningType(asset);
            String message = buildWarningMessage(asset, warningType);
            
            // 发送站内通知
            notificationService.sendInternalNotification(
                asset.getDepartmentId(), 
                message, 
                warningType
            );
            
            log.info("发送预警通知：{}", message);
        }
        
        log.info("临期资产检查完成，共发现 {} 条预警", expiringAssets.size());
    }

    /**
     * 每日清理临时文件
     */
    @Scheduled(cron = "${task.cleanup.cron}")
    public void cleanupTempFiles() {
        log.info("开始清理临时文件...");
        // 清理 7 天前的临时文件
        fileService.deleteOldTempFiles(7);
        log.info("临时文件清理完成");
    }
}
```

### 7.3 异步任务处理（线程池）

对于需要异步执行但不需要 MQ 的场景，使用 Spring 的 `@Async`：

**配置**：
```java
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(4);
        executor.setMaxPoolSize(8);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("async-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

**使用**：
```java
@Service
public class OperationLogService {

    @Async("taskExecutor")
    public void saveLogAsync(OperationLog log) {
        // 异步写入日志，不阻塞主流程
        operationLogMapper.insert(log);
    }
}
```

// 登录接口
@PostMapping("/login")
public R login(@RequestBody LoginDTO dto) {
    // 验证用户名密码
    User user = userService.getByUsername(dto.getUsername());
    if (user == null || !passwordEncoder.matches(dto.getPassword(), user.getPassword())) {

    // 登录并生成 Token

            // 权限校验（可选）
